### B03 live-hardware validation

Verified against an R307 connected to the laptop via USB-UART at
`/dev/cu.usbserial-0001` @ 57600 baud.

| Step | Result |
|---|---|
| `info` | 1000-slot library, security level 3, 128B packet size, 3 QC templates pre-loaded at slots 0/1/2 (factory leftovers, not user-enrolled) |
| `wipe` | All three QC templates cleared; `info` re-run confirmed 0 templates |
| `enroll 0` (operator-driven) | Two-capture enrollment succeeded; **characteristic = 768 bytes** (3 × 256, not the 512 the datasheet suggests — driver auto-detects via END_DATA packet); SHA-256 placeholder commitment printed |
| `search` × 2 (same finger) | match slot 0, score 134; second scan score 98 — both well above the security-3 threshold |
| `search` (different finger) | `no match found` — correct rejection |
| `search` (partial placement) | `img_to_tz confirmation=0x7` (TOO_FEW_FEATURE) — surfaced cleanly; follow-up retry helper landed in the same session |

Net: the entire enroll → search → reject pipeline works end-to-end on
real hardware. The SHA-256 we print is a placeholder; the production
flow runs the characteristic through the Pramaan fuzzy extractor +
Poseidon (lives in `/circuits`, not yet wired to this CLI).
